# OpenResty

Installs and configures **OpenResty** (Nginx with Lua) on **FreeBSD**. This role is designed to act as a secure load balancer and reverse proxy, featuring native integration with the [lego](../lego) role for SSL/TLS management and an optional **Altcha** sidecar for CAPTCHA-less challenge protection.

## Relationship with `lego`

This role is tightly coupled with the [lego](../lego) role to ensure a smooth ACME (Let's Encrypt) workflow:

1. **Shared Webroot**: OpenResty is configured to serve the ACME HTTP-01 challenge files from the directory managed by `lego` (`/usr/local/www/acme`).

2. **Certificate Pathing**: By default, OpenResty looks for SSL certificates in the `lego` certificate directory (`/var/db/lego/certificates`).

3. **Bootstrap Support**: Because OpenResty is often deployed before Let's Encrypt has issued real certificates, it relies on the self-signed placeholders generated by the `lego` role to successfully pass its configuration tests and start.

## Role Variables

### Pathing & Environment

| **Variable**             | **Default**                 | **Description**                 |
| ------------------------ | --------------------------- | ------------------------------- |
| `openresty_etc_dir`      | `/usr/local/etc/openresty`  | Configuration root.             |
| `openresty_acme_webroot` | `/usr/local/www/acme`       | Webroot for HTTP-01 challenges. |
| `openresty_ssl_cert_dir` | `/var/db/lego/certificates` | Where to find `.crt` files.     |
| `openresty_ssl_key_dir`  | `/var/db/lego/certificates` | Where to find `.key` files.     |

### Access Control

| **Variable**                  | **Default**    | **Description**                                                                     |
| ----------------------------- | -------------- | ----------------------------------------------------------------------------------- |
| `openresty_restrict_networks` | *See defaults* | List of CIDRs (e.g., Princeton internal) allowed when a vhost has `restrict: true`. |

### Vhost Configuration (`openresty_vhosts`)

Each vhost is an object in a list:

- `name`: The FQDN of the server.

- `ssl_cert`: Filename of the certificate (e.g., `app.example.com.crt`).

- `ssl_key`: Filename of the private key.

- `upstream`:

  - `name`: Unique name for the upstream group.

  - `server`: Destination address and port (e.g., `127.0.0.1:8080`).

- `restrict`: (Optional) If `true`, applies the IP allowlist.

- `access_log`: (Optional) Specific filename for the vhost access log.

### Altcha Sidecar (Optional)

This role can deploy a Node.js-based Altcha server to protect specific locations with Proof-of-Work challenges.

- `openresty_altcha_enable`: `false` (default).

- `openresty_altcha_secret`: **Required** if enabled (use a vault variable).

- `openresty_altcha_protected_locations`: List of paths to protect (e.g., `['/login', '/submit']`).

## Example Playbook

```yaml
- hosts: loadbalancers
  vars:
    openresty_vhosts:
      - name: "catalog.library.princeton.edu"
        ssl_cert: "catalog.library.princeton.edu.crt"
        ssl_key: "catalog.library.princeton.edu.key"
        upstream:
          name: "catalog_backend"
          server: "10.0.1.50:80"
        restrict: true
  roles:
    - role: lego
    - role: openresty
```

## Included Snippets

The role installs global configuration snippets in `{{ openresty_etc_dir }}/conf.d/`:

- **`acme-challenge.conf`**: Automatically included in every HTTP (port 80) vhost to route Let's Encrypt challenges.

- **`restrict.conf`**: Contains the `allow` directives for the defined `openresty_restrict_networks`.

## Altcha Integration Workflow

If `openresty_altcha_enable` is set to `true`:

1. A Node.js service is installed and managed via `rc.d/altcha_server`.

2. OpenResty is configured with an `auth_request` block.

3. Users hitting a protected location are redirected to a self-contained HTML challenge page if they do not have a valid verification cookie.
